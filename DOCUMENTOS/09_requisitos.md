# Especificación de Requisitos y Modelo — SECCAP

## 1. Requerimientos Funcionales (RF)

| ID | Requerimiento | Descripción Técnica / Funcional |
| :---: | :--- | :--- |
| **RF-01** | Autenticación Segura | El sistema debe validar el login de un operador contrastando sus credenciales contra el catálogo local protegido y asignando una sesión o token JWT en backend y memoria frontale en React. |
| **RF-02** | Pantalla Dinámica de Búsqueda | La UI proveerá un formulario compuesto con múltiples dropdowns interdependientes o *Autocomplete* para buscar perfiles por "Capacidad" y "Aptitud" (obteniendo nomenclaturas previas). |
| **RF-03** | Limitación de Resultados (Proxy) | El backend interceptará la solicitud, validará los filtros aplicados contra el **rol local (RF-04)** del usuario emisor, contactará la API institucional, y podará de la respuesta todos los atributos sensibles o no autorizados antes de conformar el objeto JSON de respuesta final al frontend. |
| **RF-04** | RBAC - Control Basado en Roles | Deben existir como mínimo perfiles Admin (crear usuarios locales), y Lectores de Rango Limitado (según reglas institucionales). |
| **RF-05** | Trazabilidad Obligatoria (Logs) | Toda petición cursada exitosamente (y las fallidas 4xx) hacia la API institucional deberá registrar en la BD local (PostgreSQL) un registro: ID_Usuario (local), timestamp, endpoint consultado, filtros aplicados y código de respuesta. |
| **RF-06** | Exportación Limitada Pantalla | Los operadores autorizados tendrán la disponibilidad de generar visualmente reportes por listas paginadas ordenadas exportables a excel/pdf básicos en base al Payload retornado, según capacidad del frontend. |

## 2. Requerimientos No Funcionales (RNF)

| ID | Requerimiento | Descripción Técnica / Infraestructura |
| :---: | :--- | :--- |
| **RNF-01** | Arquitectura Frontend | Código mantenido en React (19/Moderno), bajo TypeScript estricto, empaquetado y resuelto vía *Vite*. Estilos a través de una hoja centralizada o utilitarios como *TailwindCSS*. |
| **RNF-02** | Despliegue SPA | La UI será una Single Page Application alojada de forma estática que realiza llamadas asíncronas RESTful (mediante Axios) exclusivamente al servidor proxy. Nunca se conecta directamente a la API institucional. |
| **RNF-03** | Persistencia Relacional local | La base de datos local gestionará tablas normalizadas para autenticación, roles, sesiones, auditoría y configuraciones. No replica ni almacena datos maestros del personal. |
| **RNF-04** | Tiempo de Respuesta | El Delay provocado por el intermediario (Proxy) para mapeo no debería exceder los *350/500ms* sobre el tiempo base que provea de respuesta original la API institucional. |
| **RNF-05** | Fallbacks y Gracia UX | Ante el decaimiento de la conexión del origen institucional, el Frontend debe renderizar componentes de Fallback, alertas `Lucide React` controladas, y estado Empty sin lanzar stack traces por pantallas de navegadores comerciales (Chrome, Firefox base). |

## 3. Modelo de Dominio de Alta Jerarquía

### Componentes Lógicos
1.  **Sys_Usuario (Local):** Entidad del Software con login, asociadas a **Sys_Rol**.
2.  **Sys_Roles (Local):** Mapeo de perfiles transaccionales que definen autorizaciones y podas (truncamientos) de los payloads provenientes desde la API institucional mediante reglas.
3.  **Audit_Log (Local):** Entidad relacional vinculada directamente al ID del `Sys_Usuario` encargado del rastreo (Trazabilidad obligatoria).
4.  **Recurso_Aptitud (Modelo de Datos Proxy en Tránsito):** Entidades o Clases Tipo *DTO / Interfaces Typescript* efímeras que modelan y encasillan en tipado fuerte las estructuras impredecibles o dinámicas del retorno Institucional (JSON originario). No se guardan ni insertan en base local de la app.

### 4.4 Estructura Jerárquica e Implementación

Para asegurar un diseño consistente, la lógica jerárquica de la UI y del Backend será la siguiente:

**Tipo de formación** → **bloque específico habilitado** → **categoría** → **subcategoría / aptitud / capacitación** → **filtros transversales** (documentación, vigencia, fechas, persona, unidad).

*   Los filtros del sistema de consulta deberán ser **dependientes y jerárquicos**, no planos.
*   El filtro **Tipo de formación profesional** será el disparador principal del árbol de consulta.
*   En el bloque **Ámbito militar**, el filtro **Categoría militar** condicionará el catálogo disponible en **Capacitación o aptitud específica**.
*   En el bloque **Idioma**, el filtro **Tipo de acreditación** condicionará la forma de consulta y la procedencia del dato.
*   Los filtros **vigencia, documentación, persona, unidad y fechas** serán considerados **transversales**, siempre que la API los exponga.

## 5. Diseño Técnico y Contratos de Integración con API

> Los filtros funcionales identificados fueron traducidos a una propuesta de campos esperados en la API, definiendo para cada uno su tipo de dato, operador de búsqueda y carácter obligatorio u optativo. Esta sección servirá como insumo crítico para la etapa de diseño técnico e integración de la capa Proxy, permitiendo validar la correspondencia entre la necesidad funcional del sistema y la estructura real de datos expuesta por la API institucional. Se establece que la consulta deberá operar con filtros jerárquicos y dependientes, especialmente en el bloque de formación militar, donde la selección de una categoría condiciona fuertemente el catálogo disponible de aptitudes o capacitaciones asociadas.

### 5.5 Reglas Funcionales y Estructurales de Integración Documentadas

*   **Regla 1: Filtro dependiente.** El filtro *Capacitación o aptitud específica* no debe consultarse solo; debe elegirse ineludiblemente la categoría militar previamente.
*   **Regla 2: Filtros transversales.** Los filtros de vigencia, documentación, fecha de vencimiento, persona, unidad y estado del registro deben poder combinarse aditivamente con cualquier bloque, siempre que la API lo permita.
*   **Regla 3: Operadores correctos.** No corresponde búsqueda libre en toda la malla. Los enums/catálogos e identificadores deben operar por igualdad exacta (`=`); fechas por rangos (`>=`, `<=`); y búsquedas globales/textuales tolerar la coincidencia parcial (`LIKE` / `q`).
*   **Regla 4: Respeto a la fuente originaria.** El Proxy no debe duplicar la lógica de negocio; si la API ya trae catálogos oficiales unificados, el sistema nuevo debe limitarse a consumirlos y listarlos, sin reconstruirlos ni hardcodearlos estáticamente inútilmente.
*   **Regla 5: Comprobación de Extracción.** Esta tabla actuará como *propuesta de diseño puente*. Inevitablemente requerirá un barrido / validación real contra el contrato (Swagger/JSON) expuesto en producción por el originador de la API.

### 5.6 Ejemplos Prácticos de Construcción de Consulta (Casos de Uso)

**Caso 1: Búsqueda de Aptitud Militar Activa** (Buscando personal que tenga una aptitud específica operativa).
* Parámetros HTTP Proxy: `tipo_formacion=militar` & `categoria_militar=Aptitudes Especiales Operacionales` & `aptitud_capacitacion=En Ciberdefensa` & `compromiso_servicios_vigente=true`

**Caso 2: Búsqueda Selectiva de Idioma** (Personal con idioma inglés nivel avanzado y vigente).
* Parámetros HTTP Proxy: `tipo_formacion=idioma` & `idioma=Inglés` & `nivel_idioma=C1-C2` & `estado_vigencia=vigente`

**Caso 3: Búsqueda Administrativa de Alerta Temporal** (Buscando todo lo que vence en los próximos 90 días).
* Parámetros HTTP Proxy: `fecha_vencimiento_desde=(hoy)` & `fecha_vencimiento_hasta=(hoy + 90 días)`

### 5.7 Propuesta Tentativa de Arquitectura de Endpoints Lógicos (REST)

*   `GET /formacion/catalogos/tipos` → Devuelve civil, militar, idioma
*   `GET /formacion/catalogos/categorias-militares` → Devuelve categorías militares unificadas
*   `GET /formacion/catalogos/aptitudes?categoria=...` → Devuelve aptitudes/capacitaciones anidadas de una categoría
*   `GET /formacion/catalogos/idiomas` → Devuelve opciones de idiomas disponibles
*   `GET /formacion/catalogos/niveles-idioma` → Devuelve marco de niveles posibles
*   `GET /formacion/consulta?...` → Endpoint central. Ejecuta la búsqueda general con los correspondientes filtros sanitizados en el *Querystring*.
*   `GET /formacion/{id}` → Resuelve el detalle de un registro único del payload.
*   `GET /formacion/{id}/certificado` → Resuelve el blob/archivo respaldatorio si estuviese disponible.

## 6. Matriz Detallada de Requisitos Funcionales (Módulo de Consulta)

> A partir del relevamiento de las pantallas del sistema origen y de la estructura esperada de integración con la API institucional, se definió una matriz de requisitos funcionales estricta para el módulo de consulta de formación profesional. La misma contempla filtros jerárquicos, dependientes y transversales, incluyendo búsquedas por formación militar, idiomas, documentación respaldatoria, vigencia y datos identificatorios del personal.

### 6.3 Reglas Funcionales Derivadas del Módulo

*   **Regla RFN-01:** No se debe mostrar el filtro de *Capacitación o Aptitud específica* hasta que el usuario escoja una *Categoría Militar*.
*   **Regla RFN-02:** No se deben mostrar filtros del bloque *Idioma* cuando el tipo de formación seleccionado sea distinto al mismo (Civil o Militar).
*   **Regla RFN-03:** Los catálogos no deben cargarse manualmente o persistirse a texto en el sistema local si los mismos pueden consumirse desde endpoints en la API institucional. (Solo consumir y listar).
*   **Regla RFN-04:** La acción de Descarga Documental solo se habilitará a nivel visual en el fronten (ej. *Botón Descargar*) cuando exista respaldo disponible en respuesta (API) y el *Rol (RBAC)* posea permisos vigentes.
*   **Regla RFN-05:** Toda consulta o intento fallido de búsqueda sin resultados cruzando el proxy, debe quedar asociada ineludiblemente al usuario autenticado para fines técnicos de auditoría estricta.
